# Production Order Pricing and A4 Print Design

**Date:** 2026-07-23
**Status:** Approved design
**Scope:** Local implementation and review only; do not deploy Supabase or push Git until the user approves the local result.

## Goal

ปรับใบสั่งผลิตให้รองรับราคาต่อหน่วยซึ่งผู้ใช้กรอกเอง ระบบคำนวณจำนวนเงินต่อรายการและยอดรวมโดยอัตโนมัติ และปรับใบพิมพ์ A4 ให้ใกล้เคียงเอกสารตัวอย่าง พร้อมข้อมูลบริษัท ที่อยู่ และช่องลงนาม

## Confirmed Decisions

- ราคาต่อหน่วยเป็นข้อมูลของแต่ละรายการในใบสั่งผลิต
- ราคาต่อหน่วยต้องกรอกทุกแถวสำหรับใบใหม่และใบที่ถูกแก้ไข
- ราคาต้องมากกว่า `0` และรองรับทศนิยมไม่เกิน 2 ตำแหน่ง
- จำนวนเงินต่อแถวคำนวณจาก `จำนวน × ราคาต่อหน่วย`
- ยอดรวมสุทธิคำนวณจากผลรวมของจำนวนเงินทุกแถว
- ใบเก่าที่ยังไม่มีราคาแสดง `—` และยังเปิดดู พิมพ์ และรับเข้า stock ได้
- เมื่อแก้ไขใบเก่าที่ไม่มีราคา ผู้ใช้ต้องกรอกราคาให้ครบก่อนบันทึก
- ข้อมูลบริษัทเป็นค่าคงที่สำหรับทุกใบ ไม่แก้ไขแยกในแต่ละใบ
- ไม่แสดงเลขนิติบุคคล
- รอบแรกทำและทดสอบใน local เท่านั้น

## Data Model

### Application types

เพิ่ม `unitPrice` ในรายการใบสั่งผลิต:

- `ProductionOrderLineInput.unitPrice`: ราคาที่ผู้ใช้กรอกสำหรับใบใหม่หรือใบที่แก้ไข
- `ProductionOrderLine.unitPrice`: ราคาที่อ่านกลับจาก repository

ชนิดข้อมูลฝั่งแอปต้องสามารถแยกระหว่าง:

- ใบใหม่/ใบแก้ไข: ราคาเป็นค่าบังคับและต้อง valid
- ใบเก่า: ราคาอาจเป็น `null` เพื่อสื่อว่าไม่มีข้อมูลเดิม ห้ามแปลงเป็นศูนย์

ไม่เพิ่มฟิลด์ `lineTotal` หรือ `grandTotal` ลงในข้อมูลที่บันทึก เพราะเป็นค่าที่คำนวณได้จาก quantity และ unit price

### Money arithmetic

การคำนวณใน UI และ domain ต้องแปลงราคาเป็นหน่วยสตางค์จำนวนเต็มก่อนคูณและรวมยอด เพื่อหลีกเลี่ยง floating-point error จาก JavaScript จากนั้นจึงจัดรูปแบบกลับเป็นบาทสองตำแหน่ง

ตัวอย่าง:

- ราคา `327.50` บาท → `32750` สตางค์
- จำนวน `10` คู่ → `327500` สตางค์
- แสดงผล `3,275.00 บาท`

### Supabase

เพิ่ม migration ใหม่หลัง `202607230005_text_size_profiles.sql`:

- เพิ่ม `unit_price numeric(12,2)` ใน `public.production_order_lines`
- อนุญาต `null` สำหรับข้อมูลเดิม
- เพิ่ม constraint ให้ค่าที่ไม่ใช่ `null` ต้องมากกว่า `0`
- ปรับ `production_order_json` ให้ส่ง `unitPrice`
- ปรับ `save_production_order` ให้ใบที่บันทึกใหม่หรือแก้ไขต้องมีราคาที่ถูกต้องทุกแถว
- คงการรับสินค้าเข้า stock โดยใช้เฉพาะ variant, size และ quantity

Migration จะถูกสร้างและทดสอบใน repository แต่ห้าม deploy ไปยัง Supabase ก่อนผู้ใช้ตรวจ local และอนุมัติ

### Demo storage compatibility

Demo repository ต้อง project ข้อมูล localStorage รุ่นเก่าที่ไม่มี `unitPrice` ให้เป็น `unitPrice: null` โดยไม่ลบใบเดิมหรือทำให้ state ทั้งชุดอ่านไม่ได้ ใบใหม่ต้องบันทึกราคาในแต่ละแถว

## Form Design

ฟอร์มใบสั่งผลิตแต่ละแถวประกอบด้วย:

1. รุ่น
2. สี
3. ไซซ์
4. จำนวนคู่
5. ราคาต่อหน่วย
6. จำนวนเงิน
7. ปุ่มลบรายการ

### Price input behavior

- ใช้ช่องตัวเลขที่รองรับค่าเช่น `327` และ `327.50`
- ห้ามค่าศูนย์ ค่าติดลบ เลขที่มีทศนิยมเกิน 2 ตำแหน่ง และค่าที่เกินขอบเขตฐานข้อมูล
- ไม่เติมราคาให้อัตโนมัติจากรุ่นหรือรายการอื่น
- เมื่อผู้ใช้แก้จำนวนหรือราคา จำนวนเงินของแถวและยอดรวมเปลี่ยนทันที
- validation error ต้องผูกกับแถวที่ผิด และคงค่าที่ผู้ใช้กรอกไว้เพื่อให้แก้ไขได้

### Summary

ด้านล่างรายการแสดง:

- จำนวนรายการ
- จำนวนคู่รวม
- ยอดรวมสุทธิ

เงินแสดงด้วยรูปแบบ locale ไทยและทศนิยม 2 ตำแหน่ง เช่น `3,270.00 บาท`

### Shared inventory editor

เมนูรับเข้า นำออก แลกเปลี่ยน และฟังก์ชัน inventory อื่นไม่มีราคา จึงต้องไม่เพิ่มข้อบังคับด้านราคาใน component ที่ใช้งานร่วมกันจนกระทบ flow เหล่านั้น ฟิลด์ราคาและยอดเงินควรเป็นความรับผิดชอบเฉพาะ production-order form หรือใช้ extension point ที่เป็น optional และทดสอบว่าหน้า inventory เดิมไม่เปลี่ยนพฤติกรรม

## Detail Page

ตารางรายละเอียดใบสั่งผลิตเพิ่ม:

- ราคาต่อหน่วย
- จำนวนเงิน

ส่วนสรุปเพิ่มยอดรวมสุทธิ และคงจำนวนรายการ/จำนวนคู่รวมเดิม

สำหรับใบเก่า:

- ราคาต่อหน่วยแสดง `—`
- จำนวนเงินแสดง `—`
- หากมีทั้งรายการที่มีราคาและไม่มีราคา ห้ามแสดง grand total ที่อาจทำให้เข้าใจว่าเป็นยอดครบถ้วน ให้แสดง `—` พร้อมข้อความว่าข้อมูลราคายังไม่ครบ

หน้ามือถือแสดงข้อมูลเดียวกันในการ์ด โดยรักษาลำดับรุ่น สี ไซซ์ จำนวน ราคา และจำนวนเงินให้อ่านง่าย

## A4 Print Design

### Header layout

ใบพิมพ์ใช้ส่วนหัวสองคอลัมน์:

**ฝั่งซ้าย — ข้อมูลบริษัท**

- `STRUGGER STUDIO NO.45 CO., LTD.`
- `บริษัท สตรักเกอร์ สตูดิโอ โน.45 จำกัด`
- `เลขที่ 20 ซอย เอกชัย 80/1 ถนน เอกชัย แขวงคลองบางพราน เขตบางบอน กรุงเทพมหานคร 10150 (สำนักงานใหญ่)`
- โทรศัพท์ `084-245-5971`
- Line `@struggerofficial`
- อีเมล `Struggerofficial@gmail.com`
- ไม่แสดงเลขนิติบุคคล

**ฝั่งขวา — ข้อมูลเอกสาร**

- ชื่อเอกสาร `ใบสั่งผลิต`
- เลขที่ใบผลิต
- วันที่สั่งผลิต
- วันที่กำหนดรับ
- สถานะ

บนหน้าจอมือถือ preview สามารถเรียงข้อมูลลงมาเพื่อให้อ่านง่าย แต่ print stylesheet ต้องคงโครงสร้างซ้าย–ขวาบน A4

### Items table

คอลัมน์:

1. ลำดับ
2. รุ่นสินค้า
3. รายละเอียด สี/ไซซ์
4. จำนวน
5. ราคา/หน่วย
6. จำนวนเงิน

รูปแบบข้อมูลตัวอย่าง:

- รุ่นสินค้า: `Paris`
- รายละเอียด: `Olive / XL`
- จำนวน: `10`
- ราคา/หน่วย: `327.00`
- จำนวนเงิน: `3,270.00`

### Footer content

ใต้ตารางแสดง:

- จำนวนคู่รวม
- ยอดรวมสุทธิ
- หมายเหตุ
- ช่องลงชื่อผู้สั่งผลิตและวันที่
- ช่องลงชื่อผู้รับออเดอร์และวันที่

ใบเก่าที่ไม่มีราคาใช้ `—` ในคอลัมน์ราคาและจำนวนเงิน และยอดรวมสุทธิ ห้ามแทนด้วย `0.00`

### Printing behavior

- ปุ่มพิมพ์เรียก `window.print()` เหมือนเดิม
- รองรับเครื่องพิมพ์และ `Save as PDF`
- ผู้ใช้ยังเข้าถึง More settings ของ print dialog ได้ตาม browser
- print stylesheet ใช้กระดาษ A4, ซ่อนเมนูและปุ่มควบคุม, รักษาความอ่านง่ายด้วยหรือไม่เลือก Background graphics
- หลีกเลี่ยงการตัดแถวสินค้าและส่วนลายเซ็นข้ามหน้าเมื่อทำได้

## Company Profile

ข้อมูลบริษัทควรอยู่ใน module ค่าคงที่เดียว เพื่อให้ detail/print หรือเอกสารอื่นในอนาคตใช้ข้อมูลชุดเดียวกัน และลดข้อความซ้ำใน component

ไม่ทำหน้าตั้งค่าบริษัทใน scope นี้

## Validation and Errors

เพิ่ม validation error code สำหรับราคา เช่น `INVALID_UNIT_PRICE` และแสดงข้อความไทยที่ระบุแถวชัดเจน

ต้องตรวจ:

- ราคาเป็น number ที่ finite
- ราคามากกว่า 0
- ราคาไม่เกิน `9,999,999,999.99` ตาม `numeric(12,2)`
- ทศนิยมไม่เกิน 2 ตำแหน่ง
- ค่าที่คำนวณได้ไม่ overflow ขอบเขตที่ UI รองรับ

Supabase RPC ต้องตรวจเงื่อนไขเดียวกันเพื่อไม่เชื่อข้อมูลจาก client เพียงอย่างเดียว

## Repository Flow

1. ผู้ใช้เลือกสินค้า กรอกจำนวนและราคาต่อหน่วย
2. Form แปลงราคาเป็นค่าที่ normalize แล้วและ validate ข้อมูลทั้งใบ
3. Repository บันทึก unit price พร้อม snapshot รุ่น สี และไซซ์
4. Repository อ่านราคาเดิมกลับมาโดยไม่อ้างอิงราคาปัจจุบันจากที่อื่น
5. Selector คำนวณจำนวนคู่รวมและยอดเงินรวม
6. Detail และ print ใช้ selector/formatter เดียวกัน
7. Receive flow นำเฉพาะรายการและจำนวนไปสร้าง stock receipt เหมือนเดิม

## Testing

### Unit tests

- validation ยอมรับจำนวนเต็มและราคา 2 ตำแหน่ง
- validation ปฏิเสธราคา 0, ติดลบ, NaN, Infinity, เกินขอบเขต และเกิน 2 ตำแหน่ง
- money helper คำนวณราคาทศนิยมโดยไม่มี floating-point drift
- selector รวมจำนวนคู่และยอดเงินถูกต้อง
- selector คืนสถานะยอดไม่ครบเมื่อมี legacy line ที่ราคาเป็น `null`
- demo repository อ่านข้อมูลเก่าที่ไม่มีราคาและบันทึกข้อมูลใหม่พร้อมราคา
- Supabase mapper รองรับ `null` สำหรับข้อมูลเก่าและตรวจราคาที่ผิดรูปแบบ
- migration มี column, constraint, JSON mapping และ RPC validation ที่ต้องการ

### Component tests

- form บังคับกรอกราคาทุกแถว
- form คำนวณจำนวนเงินและยอดรวมทันที
- edit form เติมราคาเดิมกลับมา
- legacy edit บังคับกรอกราคา
- detail แสดงราคาต่อหน่วย จำนวนเงิน และยอดรวม

- print แสดงตารางราคา ยอดรวม หมายเหตุ และลายเซ็น
- ปุ่มพิมพ์ยังเรียก browser print
- receive action ยังคงส่ง quantity เดิมและไม่พึ่งราคา

## Rollout Boundary

รอบแรกสิ้นสุดที่ local preview:

- ไม่ run Supabase migration กับ remote project
- ไม่ push commit ขึ้น remote
- ไม่ trigger Cloudflare Worker build

หลังผู้ใช้ตรวจ local และอนุมัติแล้ว จึงค่อยขออนุญาตทำ migration, commit implementation และ push เป็นขั้นตอนถัดไป

## Out of Scope

- ตารางราคามาตรฐานตามรุ่น
- เติมราคาอัตโนมัติ
- หน้าตั้งค่าข้อมูลบริษัท
- ภาษีมูลค่าเพิ่ม ส่วนลด ค่าขนส่ง หรือเงื่อนไขชำระเงิน
- เลขนิติบุคคล/เลขประจำตัวผู้เสียภาษี
- เปลี่ยนวิธีรับสินค้าเข้า stock
